Workflow rapide

      

        
1Importer un fichier de logs (FortiGate ou Palo Alto CSV)

        
2Charger l'ObjDB optionnellement pour nommer les objets

        
3Sélectionner une policy dans le panneau gauche

        
4Ajuster le slider et le mode d'ancrage

        
5Exporter en CLI FortiGate, PA, CSV ou Excel

      

      
💡 ⚡ Auto-Optimize trouve automatiquement le meilleur seuil de permissivité.

    


    
    

      
      
📂 Import de logs

      
Glisse ton fichier dans la zone de dépôt. Le format est détecté automatiquement. Seuls les logs accept / allow sont traités — les deny sont ignorés.

      
Formats supportés

      

        
FortiGateSyslog texte. Champs requis : srcip, dstip, proto, dstport, action, policyid, sentbyte, rcvdbyte.

        
Palo AltoExport CSV. Champs requis : Source/Dest address, IP Protocol, Destination Port, Rule, Bytes Sent/Received.

      

      
No-return

      
Détecté quand sentbyte > 0 ET rcvdbyte = 0 — des bytes envoyés mais zéro reçu. Connexion refusée, bloquée, ou hôte injoignable.

      
⚠️ Sur les très gros fichiers (>100k lignes), le parsing peut prendre quelques secondes.

    


    
    

      
      
🎯 Sélecteur de policy

      
Les logs sont regroupés par policyid (FortiGate) ou par champ Rule (Palo Alto). Chaque policy est analysée indépendamment.

      
Si une seule policy est présente, elle est sélectionnée automatiquement. Le champ de recherche filtre la liste par nom ou ID.

      
⚠️ Si les logs PA proviennent de plusieurs firewalls, un avertissement s'affiche. Les règles ne sont pas séparées par device.

    


    
    

      
      
🗂 Object Database

      
Résout les CIDRs et ports en noms d'objets existants sur ton firewall. Sans ObjDB, des noms sont générés automatiquement (Host-10.0.0.1, TCP-8443…).

      
FortiGate — commandes

      

        
show firewall addressObjets adresse IPv4

        
show firewall address6Objets adresse IPv6

        
show firewall service customObjets service

        
show system zoneRésolution interface → zone

      

      
Palo Alto — commandes

      

        
show addressObjets adresse

        
show serviceObjets service

      

      
Formats bloc et set acceptés. Les address-group et service-group sont ignorés.

      
💡 L'ObjDB reste en mémoire navigateur — aucune donnée transmise.

    


    
    

      
      
📈 Balance Graph

      
Visualise le compromis entre nombre de règles et leur permissivité. La courbe te permet de choisir le bon équilibre d'un coup d'œil.

      

        
Courbe bleueÉvolution du nombre de règles selon la permissivité.

        
Points bleusPoints naturels de rupture — bons candidats pour le seuil.

        
Trait jauneSeuil actuellement sélectionné (slider).

        
Point vertSeuil optimal calculé par Auto-Optimize.

      

      
Clique directement sur la courbe pour déplacer le slider à ce seuil.

    


    
    

      
      
🎛 Slider de permissivité

      
Contrôle le niveau d'agrégation des flows. Score 0–100 par nœud : largeur des CIDRs (70%) + nombre de services (30%).

      
Niveaux

      

        
0–10 SurgicalRègles très précises (/32). Beaucoup de règles.

        
11–35 PreciseLégère agrégation. Bonne granularité.

        
36–65 BalancedCompromis recommandé.

        
66–80 BroadSupernets larges. Peu de règles.

        
81–100 Very BroadAgrégation maximale.

      

      
Score [N] sur les nœuds

      
Le chiffre entre crochets est le score du nœud. Un nœud est actif quand score ≤ seuil. Au-dessus, il se déplie et expose ses enfants.

    


    
    

      
      
🔒 Mode d'agrégation

      
Contrôle si FireAPG peut agréger les sources et/ou destinations en supernets (ex: /32 + /32 → /30).

      

        
AUTOFireAPG choisit l'ancrage selon les patterns comportementaux (clustering Jaccard ≥ 60%).

        
Fix SRC /32Sources toujours gardées en /32. Chaque machine source a sa propre règle.

        
Fix DST /32Destinations toujours en /32. Utile pour des règles vers des serveurs précis.

      

      
Les deux cases peuvent être cochées simultanément — aucune agrégation CIDR ne sera faite.

    


    
    

      
      
⚡ Filtre No-Return

      
Contrôle quels flows "sans réponse" sont inclus dans l'arbre. Le filtre agit avant la construction — il reconstruit l'arbre entier.

      
Tags de destination colorés

      
Les destinations avec du trafic no-return sont entourées d'une bordure colorée selon la catégorie :

      

        
🔴 NR-TCPTCP sans réponse — connexion refusée, bloquée ou hôte injoignable.

        
🟠 NR-UDP?UDP no-return sur port inhabituel. Potentiellement suspect.

        
🟡 NR-UDPUDP one-way normal (DNS, NTP, Syslog…). Comportement attendu.

      

      
Bouton × — Exclusion d'export

      
⚠️ Le × n'est pas un filtre d'arbre — il exclut l'IP de l'export uniquement. L'arbre n'est pas reconstruit.

      
Quand tu cliques × sur un tag destination :

      

        
Effet visuelLe tag passe en grisé barré. Le bouton devient ↩ pour réactiver.

        
Effet exportL'IP est retirée des destinations dans le CLI et le CSV générés.

        
Effet arbreAucun — le nœud reste visible. Si toutes les destinations sont exclues, le nœud affiche le badge ⚠ et sera absent de l'export.

      

      
Cas d'usage : une IP a 90% de flows normaux + quelques NR. Tu veux la garder dans l'analyse mais l'exclure de l'export pour un nœud précis, sans toucher aux autres nœuds ni reconstruire l'arbre.

      
💡 Pour retirer les flows NR de tout l'arbre, utilise les boutons "Excl. 🔴 TCP" ou "Excl. non-normal" dans le panneau gauche.

      
Modes de filtrage (panneau gauche)

      

        
Show allTous les flows dans l'arbre. NR signalés visuellement uniquement.

        
Excl. 🔴 TCPRetire les flows TCP NR avant construction. L'arbre est reconstruit.

        
Excl. non-normalRetire TCP NR + UDP suspect. Garde uniquement 🟡 UDP normaux.

      

    


    
    

      
      
🌲 Rule Expansion Interface

      
L'arbre central = les règles candidates. Chaque nœud = une règle agrégée. Les nœuds ＋/－ ont des enfants (règles plus granulaires).

      
Lecture d'un nœud

      

        
SRC ↳Sources IP + interface/zone d'origine.

        
DST ↳Destinations. Tags colorés = statut NR.

        
SVCServices. Nom ObjDB si disponible.

        
N fl.Nombre de flows couverts.

        
[42]Score de permissivité (0–100).

      

      
Actif vs grisé

      
Nœud actif = sera exporté. Nœud grisé = score dépasse le seuil, ses enfants actifs le remplacent dans l'export.

    


    
    

      
      
🏷 Badges des nœuds

      
Survoler un badge affiche un tooltip avec le détail (liste des services ou IPs concernés).

      
🔴 NR-TCPTCP no-return dans ce nœud.

      
🟠 NR-UDP?UDP suspect (port non standard).

      
🟡 NR-UDPUDP one-way normal (DNS, NTP…).

      
🔵 SNATSource NAT détecté. Tooltip : IPs translatées.

      
⊕ MergedNœud issu d'un merge manuel. ✕ pour annuler.

      
✓ TraitéNœud marqué comme traité/validé.

      
⚠ EmptyToutes les destinations exclues via ×. Absent de l'export.

    


    
    

      
      
⊕ Merge manuel

      
Fusionne N nœuds sélectionnés en une seule règle (union des sources, destinations et services).

      
Sélectionne des nœuds avec les cases à cocher, puis clique ⊕ Merge dans la toolbar.

      
Règles de merge

      
Les /32 (hôtes) sont toujours conservés distincts — seuls les réseaux (/1–/31) peuvent être agrégés en supernet. Les interfaces sont toutes conservées (multi-intf dans le CLI généré).

      
Clique ✕ sur le badge ⊕ Merged pour annuler et retrouver les nœuds originaux.

    


    
    

      
      
✓ Marquage Traité

      
Marque des nœuds comme traités (validés, décidés). Le nœud se replie et affiche un badge vert.

      
Sélectionne des nœuds avec les cases à cocher, puis clique ✓ Traité. Une popup propose d'ajouter une note libre (ex : "Validé — monitoring interne").

      
Impact sur les exports

      
Par défaut, les nœuds traités sont exclus de l'export (export "prêt à déployer"). Dans Export Hub, le toggle Inclure les traités permet de les réintégrer avec annotation.

      
💡 Le compteur [N / total traités] dans la topbar indique ta progression.

    


    
    

      
      
📦 Export Hub

      
Génère les règles depuis les nœuds actifs. Si des nœuds sont cochés, seuls ceux-là sont exportés. Sinon, tous les nœuds actifs sont inclus.

      

        
FortiGate CLIBlocs config firewall address/service/policy. Objets créés si absents de l'ObjDB.

        
Palo Alto CLICommandes set address/service/security. Multi-zone supporté.

        
CSVUne ligne par règle. Colonnes ObjDB et zones si disponibles.

        
ExcelCSV + feuilles "Raw Flows" et "No-Return Flows".

      

      
Options

      

        
🔴 Add disabledRègles générées avec status disable — à importer sans activer.

        
Inclure traitésRéintègre les nœuds "Traité" dans l'export avec annotation.

      

    


    
    

      
      
🔍 Flow Lens

      
Explorateur de flows bruts. Navigation par Source → Destination → Service (ordre libre).

      
Navigation

      
Clique sur une entrée pour "driller". Un fil d'Ariane montre ta position — clique dessus pour remonter. L'ordre des dimensions est personnalisable (boutons SRC / DST / SVC).

      
Highlight REI

      
Au niveau le plus profond, clique sur une entrée pour mettre en surbrillance le nœud REI correspondant. L'arbre s'expanse automatiquement si le nœud est collapsé.

      
💡 La recherche filtre sur la valeur brute ET sur le nom ObjDB simultanément.

    



    
    

      
      
📊 Traffic Summary

      
Statistiques globales sur la policy sélectionnée — calculées sur l'ensemble des flows accept.

      

        
SourcesNombre d'IPs sources distinctes observées.

        
DestinationsNombre d'IPs destinations distinctes observées.

        
ServicesNombre de combinaisons proto/port distinctes.

        
Total flowsNombre total de lignes de log acceptées pour cette policy.

      

      
Ces chiffres reflètent les flows bruts, avant agrégation. Le nombre de règles générées dans l'arbre sera toujours inférieur.

    


    
    

      
      
🔵 Badges NAT

      
Indique qu'au moins un flow de ce nœud a fait l'objet d'une translation d'adresse (NAT). Survoler le badge affiche les IPs et ports traduits.

      

        
🔵 SNATSource NAT — l'IP source a été translatée (ex : masquerading, PAT). Tooltip : IP/port source après translation.

        
🟣 DNATDestination NAT — l'IP destination a été translatée (ex : port forwarding). Tooltip : IP/port destination réels.

        
🔵 SNAT+DNATLes deux types de NAT sont présents dans ce nœud.

      

      
Le NAT est détecté depuis les champs transip / transport (FortiGate) ou colonnes NAT (Palo Alto). Si ton export ne contient pas ces colonnes, les badges n'apparaissent pas.

    


    
    

      
      
💡 Security Insights

      
Analyse automatique affichée après chargement d'une policy. Les insights sont informatifs — ils ne modifient pas l'arbre ni les exports.

      

        
🔴 TCP no-returnFlows TCP sans réponse. Pourcentage affiché. Peut indiquer des scans, des hôtes injoignables ou des règles permissives inutiles.

        
🟠 UDP suspectUDP no-return sur ports non standards. Liste des ports concernés.

        
🌍 Pays destinationPays vers lesquels du trafic a été observé (champ dstcountry).

        
🧠 ClustersSources regroupées par similarité comportementale (≥ 60% de destinations+services communs). Indique combien de fusions automatiques ont eu lieu.

        
🚫 IPs excluesIPs exclues manuellement via × dans l'arbre. Cliquer sur l'insight pour voir la liste et les réactiver.

      

    



    
    

      
      
⚡ Auto-Optimize

      
Trouve automatiquement le seuil de permissivité qui minimise un score combiné : nombre de règles × permissivité moyenne × nombre de services distincts.

      
Comment ça fonctionne

      
FireAPG parcourt tous les seuils de 0 à 100, calcule le nombre de règles actives et leur score moyen à chaque niveau, puis sélectionne le point d'équilibre optimal. Le résultat est affiché comme point vert sur le Balance Graph.

      

        
RésultatLe slider se positionne au seuil optimal. L'arbre est mis à jour.

        
Point vertVisible sur le graphe — correspond au seuil trouvé.

        
OverrideTu peux ensuite bouger le slider manuellement pour t'écarter de l'optimal si besoin.

      

      
💡 Auto-Optimize respecte les manualOverride des nœuds que tu as manuellement développés ou repliés.
